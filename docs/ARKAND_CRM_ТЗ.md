## 1. Что строим
Единая система из **8 подсистем** на одной БД: Застройщик, Проектная компания, Бетонный завод,
Щебёночный завод, Снабжение, Финансы, Надстройка владельцев, Отдел проверки (read-only).
Данные вводятся один раз, синхронизируются мгновенно (WebSocket).

## 2. Стек (жёстко)
- Frontend: React+TS (Vite) → Vercel; TanStack Query v5; Zustand; нативный WebSocket+reconnect;
  TailwindCSS + CSS-переменные; React Router (lazy); react-hook-form + zod.
- Backend: Python+FastAPI → Railway; WebSocket + Redis Pub/Sub; PostgreSQL; SQLAlchemy+Alembic;
  Pydantic v2; JWT (access+refresh), argon2/bcrypt.

## 3. Архитектура — строгий FSD
`app → processes → pages → widgets → features → entities → shared`. Импорт только вниз.
Между слайсами одного слоя — нельзя. Публичный API только через `index.ts`.
Слайс: `ui/ model/ api/ lib/ index.ts`.

## 7. Сквозные бизнес-правила (во ВСЕХ модулях)
1. **Append-only** — проведённое нельзя менять/удалять, только сторно. Автор+время. История неизменяема.
2. **Обратные потоки** — `отменён/возвращён/выполнен_частично/брак` = штатные статусы.
3. **Заморозка значений** на дату (цена/смета/рецептура/курс).
4. **Статусы денег/товара** — деньги: `в_кассе → в_пути → принято`; доход: `аванс → отработано → выручка`;
   сырьё: `на_складе → в_производстве → в_готовом → отгружено`.
5. **Единые справочники** — номенклатура/единицы/контрагенты. Свободный текст запрещён.
6. **Разделение полномочий** — кто отгрузил ≠ кто принял деньги (разные логины, проверка на сервисе).
7. **Аудит-лог** — неизменяемый (кто/что/когда/до/после), read-only для отдела проверки.

## 8. Роли и доступы (RBAC)
Permission = `(resource, action, scope)`. Роль = набор permissions. Проверка **на бэке**.
- **Сохиб** — гл. финансист, полный доступ. **Ифтихор** — суперадмин, полный. **Довуд** — проектная зона.
- Крупные расходы — согласие **всех троих**. Порог задают владельцы.
- Защита владельцев: отключить доступ владельцу — только с согласия другого владельца.
- Разделы **Сотрудники/Аналитика/Календарь** — только Сохиб и Ифтихор.
- scope: `all | own_business:<id> | own_records | read_only`.

## 9. Модули
- **9.1 Застройщик**: Объект, СметаОбъекта, ЗаявкаНаМатериалы, НакладнаяПриёмки, ДвижениеСклада, ФинансыОбъекта, Инвентаризация. Смета заморожена; недобор/перебор — видимая операция.
- **9.2 Проектная (Довуд)**: Клиент, Заявка, Бриф, Договор(50/30/20), Проект, Этап, ОбъектНадзора, Обход, ОплатаНадзора, Геодезия. Аванс = обязательство, не выручка. Регистрация клиента обязательна.
- **9.3 Бетонный завод**: Заказ, ПроизводственноеЗадание, Рецептура1м3, Себестоимость, ТалонОтгрузки, Доставка, СкладСырья, Техника, РемонтТО, ПаспортКачества. Отгрузка ≠ касса. Бартер по прайсу.
- **9.4 Щебёночный завод**: ПродукцияФракция, ПроизводствоСмена, ВыпускФракции, Заказ, ТалонСамосвал, РасходСолярка, Мощность, СебестоимостьЕдиницы, Техника. Три роли (кассир/снабженец/финансист) разделены.
- **9.5 Снабжение**: ЗаявкаНаЗакупку, Поставщик, КП, Закупка, Лимит, Оприходование, СкладВид, Инвентаризация, ЗаписьДолга, РеестрКонтрагентов. Лимиты меняют владельцы; межбизнес-передача = долг; контроль дробления.
- **9.6 Финансы**: Проводка, Касса, ЛимитКассы, Инкассация, Взаиморасчёт, Долг, Бартер, Зарплата, Табель, Аванс, ЗакрытиеПериода. append-only; аванс ≠ выручка; расход привязан к основанию.
  - **9.6.1 Инкассация (точно так):** `в_кассе → в_пути → принято` + `расхождение`.
    1. Кассир жмёт «Инкассация» → система показывает расчётный остаток (справочно).
    2. Кассир вводит **фактическую** сумму (обязательный шаг).
    3. Расхождение → отдельная видимая операция `недостача/излишек` (привязка к смене+кассиру).
    4. Деньги → `в_пути`.
    5. Получатель (директор) под своим логином подтверждает принятую сумму.
    6. Совпало → `принято`; нет → `расхождение` (видно обоим и отделу проверки).
    Остаток только вычисляется; двусторонняя (передал ≠ подтвердил); задним числом — только сторно.
- **9.7 Надстройка владельцев**: консолидация, согласование крупного (все трое), пороги/лимиты,
  Сотрудники/Аналитика/Календарь (Сохиб/Ифтихор), управление доступом (защита владельцев), задачи, KPI.
- **9.8 Отдел проверки (read-only)**: ПланПроверок, АктПроверки, Замечание, СверкаКассы/Рецептуры/Талонов,
  ЭскалацияНарушения. Только чтение; эскалация — всем троим, минуя зону нарушителя.

## 11. API (примеры)
REST для CRUD, WS для реалтайма. Все мутации проверяют permissions и пишут в audit_log.
Никаких PUT/DELETE по проведённым деньгам/складу — только `reverse`. Списки — пагинация+фильтры.
Ошибки — единый формат `{code, message, details}`.
- `POST /api/inkassaciya/start {cash_id} -> {calc_amount}`
- `POST /api/inkassaciya/{id}/fact {fact_amount} -> {status, discrepancy}`
- `POST /api/inkassaciya/{id}/accept {accepted_amount} -> {status}` (другой пользователь)
- `POST /api/orders {order} -> {id}` (рецептура замораживается)
- `POST /api/orders/{id}/reverse {reason} -> {reversal_id}`
- `POST /api/supply/requests {items, business_id} -> {id}`
- `POST /api/approvals/{id}/vote {owner, decision} -> {result}` (нужны все трое)
- `POST /api/employees/{id}/access {permission, active} -> {ok}` (владельца — с согласия другого)
- `GET /api/analytics/summary?period=... -> {by_business, total}` (Redis-кеш)

## 12. WebSocket
Один эндпоинт `/ws`; после подключения — JWT-авторизация и подписка по правам.
Каналы: `business:<id>`, `cash:<id>`, `supply`, `finance`, `owners`, `audit`, `employee:<id>`.
Мутация → Redis Pub/Sub → рассылка подписчикам. Событие: `{channel, type, payload, at}`.
Клиент патчит/инвалидирует Query-кеши. Обязательно: reconnect+backoff, дедуп, ресинк после реконнекта.
